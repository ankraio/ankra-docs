# More Reliable Apply for Large Manifest Stacks

Applying a stack with many manifests at once — and reconciling stacks on clusters with lots of Custom Resource Definitions — is now far more reliable. Manifests are given more time to apply before they're considered stalled, and the agent no longer re-runs expensive API discovery on every single apply, so a large stack reconciles cleanly instead of failing partway through with a timeout.

The change applies automatically: existing stacks pick it up on their next deploy with no action required. Any per-manifest timeout you set previously is honoured as long as it's at least the new, more generous default, and custom timeouts above the default are kept as-is.

[Manifests →](/essentials/manifests)

# Full OVH Cluster Management from the CLI

Everything you can do to an OVH cluster in the web UI is now available in the Ankra CLI. You can stop a cluster to release its compute while keeping its configuration, start it again (optionally bringing up just the control plane first), view SSH access details, manage the SSH keys attached to a cluster, and inspect or change the control plane and individual nodes.

```bash
ankra cluster ovh stop <cluster_id>
ankra cluster ovh start <cluster_id> --scope control_plane
ankra cluster ovh access-info <cluster_id>
ankra cluster ovh ssh-keys get <cluster_id>
ankra cluster ovh ssh-keys set <cluster_id> --ssh-key-credential-ids <id>,<id>
ankra cluster ovh control-plane get <cluster_id>
ankra cluster ovh nodes list <cluster_id>
ankra cluster ovh node-group add <cluster_id> --name gpu --instance-type b2-30 --labels tier=gold --taints dedicated=gpu:NoSchedule
```

`ankra cluster ovh access-info` prints the gateway and control plane IPs together with ready-to-paste `ssh -J` jump and Kubernetes API port-forward commands, and `node-group add` now takes `--labels` and `--taints` so a pool starts with its scheduling rules in place.

[OVH Clusters →](/essentials/ovh-clusters)

# Discover OVH Regions and Manage Node Labels & Taints from the CLI

Picking an OVH region for a new cluster no longer means guessing. `ankra cluster ovh regions --credential-id <id>` lists the regions your OVH project can actually deploy in, so `ankra cluster ovh create --region ...` succeeds the first time instead of failing partway through network setup on a region the project never enabled.

You can also manage node scheduling directly from the CLI:

```bash
ankra cluster ovh regions --credential-id <credential_id>
ankra cluster ovh node-group labels <cluster_id> <group> --labels workload=realtime,tier=gold
ankra cluster ovh node-group taints <cluster_id> <group> --taints dedicated=realtime:NoSchedule
```

Labels and taints are applied to every node in the group; passing an empty value clears them, and a taint effect defaults to `NoSchedule`.

# Keep the CLI Up to Date — and Roll Back — with `ankra upgrade`

The Ankra CLI can now update itself in place. `ankra upgrade` resolves the latest release, verifies the download against its published SHA-256 checksum, and atomically swaps the running binary — no need to re-run the install script.

```bash
ankra upgrade                       # upgrade to the latest release
ankra upgrade --check               # is a newer release available?
ankra upgrade --version v0.2.5      # install an exact release
ankra upgrade --version 0.1.9 --yes # downgrade / roll back to an older release
```

Pin an exact release with `--version` (the leading `v` is optional) to upgrade **or** downgrade: a pinned version is installed whether it is newer, older, or the same as the one you are running, so the same command rolls back a bad release. Downloads are checksum-verified before the swap, and an upgrade for a release with no published checksum aborts rather than installing an unverified binary (override with `--allow-unverified`). Opt into release candidates any time with `ankra config beta enable`.

[Ankra CLI →](/integrations/ankra-cli)

# Stage Cluster Changes as Drafts from the CLI

`ankra cluster draft -f cluster.yaml` stages every stack in an ImportCluster file as a reviewable draft instead of applying it. It runs the same fast local checks as `--dry-run`, then saves each stack as a resource draft you can review, edit, and deploy from the stack builder — nothing is deployed by the command itself. It's the CLI path to the same draft workflow the UI uses, so you can propose changes from Git or a pipeline and have someone approve them before they go live.

Stacks that already match the cluster's desired state are reported as "no changes" instead of creating empty drafts, and if the cluster doesn't exist yet it's imported first (drafts can only attach to an existing cluster).

```bash
ankra cluster draft -f cluster.yaml
```

[Ankra CLI →](/integrations/ankra-cli)

# Validate Cluster Configuration Before You Apply It

`ankra cluster validate -f cluster.yaml` checks an ImportCluster definition without changing anything. It runs the same fast local checks as `--dry-run` (structure, referenced-file YAML, and the parent/dependency tree) and then asks the Ankra API to run the checks that need server-side data:

- **Chart existence** — every addon's chart must exist in a Helm registry connected to your organisation.
- **Plaintext secrets** — Kubernetes `Secret` manifests and addon values that aren't SOPS-encrypted are flagged so secrets don't land in Git in the clear.
- **Parent references** — with `--cluster <id>`, dependencies are resolved against that cluster's already-deployed resources, not just the file.

Warnings are printed without failing; pass `--strict-secrets` to make plaintext secrets a hard error. The command exits non-zero when it finds errors, so it drops straight into a CI pipeline as a pre-apply gate.

```bash
ankra cluster validate -f cluster.yaml
ankra cluster validate -f cluster.yaml --strict-secrets
ankra cluster validate -f cluster.yaml --cluster <cluster_id>
```

[Ankra CLI →](/integrations/ankra-cli)

# Watch Cluster Operations and Job Status Live from the CLI

The Ankra CLI can now poll operations and their jobs in place, so you can follow a deploy or reconcile to completion without re-running a command in a loop.

```bash
# Refresh the operations table until everything reaches a terminal state
ankra get operations --watch

# Follow a single operation, refreshing every 3 seconds
ankra get operations <operation_id> --watch --interval 3s

# Track the per-job status for an operation as it runs
ankra get jobs <operation_id> --watch
```

`--watch` redraws the screen on each poll (default every 5 seconds, tunable with `--interval`) and stops automatically once the operation — or every operation in the list — finishes, so the command exits cleanly instead of spinning forever. The new `ankra get jobs <operation_id>` command shows the operation summary plus each job's status, created, and updated times, and is watchable the same way. Statuses are colorized (green success, red failure, yellow running) to make in-progress work easy to spot at a glance.

# Removed: Add-on Subscriptions

Add-on subscriptions have been retired. Add-ons are now managed directly on the cluster or stack that uses them — through your GitOps repository, `ankra cluster apply`, or the cluster editor — which keeps a single, declarative source of truth for what runs where, rather than a separate subscription that has to be kept in sync.

The legacy add-on subscription endpoints now return `410 Gone`. If you relied on subscriptions, move those add-ons into the relevant stack definition; existing deployed add-ons are unaffected.

# Manage Variables at Org, Cluster, and Stack Scopes from the CLI

The Ankra CLI now ships full CRUD for the variables that get substituted into stack manifests and addon values at deploy time, at all three scopes — organisation, cluster, and stack.

```bash
# Org-wide, available to every cluster
ankra org variables set DB_HOST db.example.com --description "Primary DB"

# Per-cluster, shadows org variables on that cluster
ankra cluster variables set DB_HOST db.prod.example.com --cluster prod

# Per-stack, the most specific scope
ankra cluster stacks variables set demo-web-app FEATURE_FLAG enabled
```

Each scope has `list` / `get` / `set` / `delete`. `set` is an upsert — it creates or updates — and values can be read from stdin with `-`, so secrets can be piped in from a vault or password manager. Resolution at deploy time is **stack > cluster > organisation**, so a more specific scope shadows less specific ones for the same variable name. List/get support `-o json|yaml` for scripting, and `delete` prompts for confirmation (`--yes` to skip).

[CLI Organisation Variables →](/integrations/ankra-cli#org-variables) · [Cluster Variables →](/integrations/ankra-cli#cluster-variables) · [Stack Variables →](/integrations/ankra-cli#cluster-stacks-variables)

# SOPS Encrypt and Decrypt Directly Against Live Clusters

`ankra cluster encrypt` and `ankra cluster decrypt` now work directly against a live cluster — no local `cluster.yaml` needed. Cluster mode fetches the current manifest or addon values, runs them through the SOPS API, and (for encrypt) pushes the result back via the same partial-stack PATCH used by `manifests upgrade` / `addons upgrade`, updating `encrypted_paths` automatically.

- **Encrypt:** `ankra cluster encrypt manifest db-secret --key data.password`, `ankra cluster encrypt addon --name grafana --key adminPassword --cluster prod`
- **Decrypt:** `ankra cluster decrypt manifest db-secret`, `ankra cluster decrypt addon --name grafana --cluster prod`

The existing `-f cluster.yaml` file mode is unchanged and remains for GitOps workflows where the source of truth is on disk. The two modes are mutually exclusive; cluster mode is the default. A new `decrypt addon` brings the addon variant to parity with manifests.

[CLI SOPS Encryption →](/integrations/ankra-cli#sops-encryption)

# Manage Dependency Parents, Reads, and Manifest Deletes from the CLI

The Ankra CLI now covers the rest of stack CRUD without hand-editing `ImportCluster.yaml`.

- **Edit dependency parents** that control deployment ordering: `ankra cluster addons upgrade infisical --add-parent name=infisical-ns,kind=manifest`. `--remove-parent` and `--set-parent` are available on both `addons upgrade` and `manifests upgrade`.
- **Read current content**: `ankra cluster addons values <name>` prints an addon's Helm values and `ankra cluster manifests get <name>` prints a manifest's YAML — both ready to pipe to a file, edit, and re-apply.
- **Delete a manifest**: `ankra cluster manifests delete <name>` disconnects a manifest from its stack and removes its resources from the cluster, resolving the owning stack automatically and prompting for confirmation.

# Run CLI Commands Against Any of Your Organisations

The Ankra CLI now has a global `--org` flag (and an `ANKRA_ORG` environment variable) so you can run a single command against any organisation you belong to without switching your selected organisation first. Pass an organisation name or ID:

- `ankra --org "Acme Corp" cluster list` runs just that command against another organisation.

The override is applied per request — it never changes the organisation `ankra org switch` selected, so scripts and parallel sessions targeting different organisations no longer step on each other. You must be an active member of the organisation you target.

# Install Curated AI Agent Skills from the CLI

`ankra skills` installs the curated Ankra Agent Skills into your Cursor/Claude skills directory, so your AI agent follows Ankra's recommended practices for the CLI, ImportCluster YAML, stacks/addons, GitOps, CI/CD, SOPS secrets, Helm registries, observability, alerts, Terraform, and cloud clusters. The skills are embedded in the CLI binary, so installation works offline and is versioned with the release.

```bash
ankra skills list                  # list available skills (marks installed ones)
ankra skills install               # install all into ~/.cursor/skills (personal)
ankra skills install --project .   # install into ./.cursor/skills (project)
ankra skills install ankra-gitops  # install only named skills
ankra skills uninstall             # remove all Ankra skills
```

Use `--project <dir>` for a per-project install (`./.cursor/skills`) and `--force` to overwrite existing skills. This is separate from `ankra openclaw skill`, which generates a SKILL.md describing one of your own clusters; `ankra skills` installs the static, curated skill set.

# Edit Cluster Topology While Stopped

Stopped cloud clusters (Hetzner, OVH, UpCloud) now expose the full Nodes settings tab. While the cluster stays offline you can:

- Switch the control plane between **1** and **3** controllers (only 1 or 3 — etcd needs an odd number of voting members to form quorum).
- Change the **controller instance type**. Because the cluster is recreated fresh on next start, downgrades are safe here — there is no live disk resize.
- Add and remove **node groups**, scale them, change their worker instance type, and edit labels and taints.

Changes are saved against the stopped topology and applied the next time you start the cluster from the General settings tab. Topology edits now also serialize against each other and against start/stop, so two operators clicking at the same time can't end up with a half-applied change.

# CLI Surgical Upgrades for Addons and Manifests

The Ankra CLI now ships `ankra cluster addons upgrade` and `ankra cluster manifests upgrade` for in-place changes, removing the need to hand-edit a full `ImportCluster.yaml` to roll a chart version, tweak Helm values, or rotate a manifest.

- **Mutate a single Helm values field** with helm-style `--set` (dotted paths, array indexing): `ankra cluster addons upgrade website --set image.tag=1.0.146`
- **Replace the whole values document** with `--values-from-file`.
- **Update a manifest** from a file or stdin with `ankra cluster manifests upgrade <name> --from-file <path>`.
- **Atomic + GitOps friendly**: every invocation calls the partial-stack endpoint, locks the cluster, and produces a single git commit when gitops is enabled.
- **`--dry-run`** prints a before/after preview without writing; supports `-o json|yaml` for CI gating.

[CLI Changelog →](/integrations/ankra-cli-changelog)

# In-Place `--set` Edits for Manifests

`ankra cluster manifests upgrade` now accepts helm-style `--set`, `--set-string`, and `--set-file` to patch a single path inside a manifest's Kubernetes YAML — so bumping a Deployment image tag from CI no longer means rewriting the whole file.

- **Patch one path**: `ankra cluster manifests upgrade web --set 'spec.template.spec.containers[name=app].image=nginx:1.27'`
- **Multi-document manifests**: pick the document to edit with `--target-kind` and `--target-name`.
- **Field selectors everywhere**: both manifest and addon `--set` paths can address a list item by a stable field — `containers[name=app]`, `env[name=LOG_LEVEL]` — instead of a fragile numeric index. A selector that matches nothing fails loudly rather than guessing, and numeric indexes still work.

`--set*` mutate the existing content and are mutually exclusive with `--from-file` / `--manifest -`, which replace it.

## Hetzner: Resilient Cluster Updates

Hetzner cluster update jobs are now self-healing. The reconciler persists install-time addresses (`bind-address`, `advertise-address`, `node-ip`) into `/etc/rancher/k3s/config.yaml` so k3s always restarts bound to the correct private IP, and version bumps now swap the k3s binary in place instead of re-running the upstream installer — which previously rewrote the systemd unit and wiped `K3S_URL` / `K3S_TOKEN` on joined nodes. Clusters that had drifted into authentication-error loops after an upgrade recover on the next reconcile.

## Offline Cloud Clusters: Cleaner Reconciliation

When a Hetzner, OVH, or UpCloud cluster is in the **stopped** state (after Delete Kubernetes), the reconciler now only acts on provider-owned resources — servers, networks, SSH keys, the k8s control plane. Addons, manifests, and other in-cluster resources are held until the cluster is back online, so you no longer see repeated failures for resources that have no API server to talk to.

[Hetzner Cluster Lifecycle →](/essentials/hetzner-clusters#cluster-lifecycle)

## Operations Page: Live Updates

The cluster **Operations** page now refreshes itself in the background. Newly-triggered operations appear without a manual reload, and the status, duration, and finish time of in-flight operations update as they progress. The page polls faster while something is actively running and slows down once everything is idle, so you get near-real-time feedback while a deploy is happening without hammering the backend when nothing is going on.

</Update>

# Cluster Lifecycle: Delete Kubernetes & Scale Back Up, Multi-Stack Cloning

## Delete Kubernetes & Scale Back Up

Cloud provider clusters (Hetzner, OVH, UpCloud) now support a **Delete Kubernetes** action that destroys all cloud infrastructure — servers, networks, SSH keys — while keeping the cluster registered in Ankra. The cluster moves to a stopped state and you stop incurring infrastructure charges.

When you're ready to bring the cluster back, simply **add a node group**. Ankra automatically restores the full control plane infrastructure (bastion, network, SSH keys, control plane servers), installs Kubernetes, creates your new worker nodes, and begins reconciling stacks and addons. The restored cluster gets entirely fresh infrastructure with new servers and IPs.

Previously existing worker node groups are not restored — the node group you add is the only worker pool. Add additional node groups as needed after the cluster is online.

The **Terminate Cluster** action permanently deletes all infrastructure and removes the cluster from Ankra entirely. This is irreversible.

Both actions are available in cluster **Settings → Danger Zone**.

[Hetzner Cluster Lifecycle →](/essentials/hetzner-clusters#cluster-lifecycle) | [Cluster Settings →](/essentials/cluster-settings#danger-zone)

## Clone Multiple Stacks to Another Cluster

You can now select multiple stacks and clone them all to another cluster in a single operation. On the **Stacks** page, click any stack card's avatar to start selecting, or click the **Clone Stacks** button next to the search bar to select all. A floating action bar appears showing the selection count with a **Clone to Cluster** button.

The clone dialog shows a live progress bar as each stack is cloned sequentially. Name conflicts are auto-resolved with suffixes. After completion, click **Go to Target Cluster** to review the drafts. Cross-organisation cloning is fully supported.

This is useful for replicating an entire cluster's stack configuration to a new environment, promoting multiple stacks from staging to production, or setting up disaster recovery clusters.

[Clone Stacks Guide →](/guides/clone-stack#clone-multiple-stacks-at-once)

## Kubernetes Node Cordon, Uncordon & Drain

The **Kubernetes → Nodes** view now supports full node lifecycle management for workload migration scenarios. Cordon, uncordon, and drain nodes from both the node detail page and the node list with multi-select bulk actions.

**Cordon & Uncordon**: Mark nodes as unschedulable or restore scheduling. Available as a single-node action in the node detail header or as a bulk action by selecting multiple nodes in the list. Cordoned nodes display a "SchedulingDisabled" label.

**Drain**: Cordon a node and evict all running pods in one operation — equivalent to `kubectl drain --ignore-daemonsets --delete-emptydir-data`. Real-time progress tracking shows each phase: cordoning, fetching pods, and evicting pods. Available for single nodes and in bulk.

**Workload Migration**: Use cordon and drain together to safely move workloads between node groups. Cordon old nodes to stop new scheduling, drain them to evict existing pods, and let Kubernetes reschedule onto your new node group.

[Kubernetes Nodes →](/essentials/kubernetes-workloads#nodes)

## Node Groups Per-Group Instance Types, Labels & Taints

A new **Nodes** tab in cluster settings lets you manage node groups for Hetzner, OVH, and UpCloud clusters. Each node group has its own instance type, node count, Kubernetes labels, and taints giving you fine-grained control over your worker fleet.

**Full Lifecycle Management**: Add, scale (0–100 nodes), upgrade instance type, edit labels/taints, and delete node groups from the UI, CLI, or API. Node groups can be defined at cluster creation time in the wizard or API.

**Instance Type Upgrades**: Upgrade any group to a larger instance type. Each node is powered off, resized, and powered back on. Downgrades are not possible due to Hetzner disk resize limitations. **Labels & Taints**: Attach Kubernetes node labels and taints to node groups. Labels and taints are applied to all nodes in the group and can be edited at any time. Useful for workload scheduling with nodeSelectors, affinities, and tolerations.

**No Online Requirement**: Node group operations work regardless of cluster state. No need to wait for the cluster to be online.

**CLI Support**: New `ankra cluster {hetzner|ovh|upcloud} node-group` commands for list, add, scale, upgrade, and delete.

## AI Insights Overhaul
