AWS credentials let Ankra read your account to estimate the infrastructure cost of your clusters and sync cloud inventory, and to provision and manage [Amazon EKS](/guides/eks-clusters). You can connect in one of two ways:

<CardGroup cols={2}>
  <Card title="IAM role (recommended)" icon="shield-halved">
    A one-click CloudFormation stack creates a role Ankra can assume, secured with a unique external ID. No long-lived keys to store or rotate.
  </Card>
  <Card title="Access keys" icon="key">
    Provide an access key ID and secret for an IAM user. Simpler to set up, but you own key rotation.
  </Card>
</CardGroup>

## Access levels

The IAM role method offers two access levels. Pick the level when you start the wizard - it determines which CloudFormation template you launch:

| Access level | What Ankra can do | Choose it when |
|---|---|---|
| **Read-only (cost & inventory)** | Read pricing and Cost Explorer data, describe EC2 instance types and regions, list and describe EKS clusters. Never provisions or modifies AWS resources. | You only want cost estimates and inventory. |
| **Provisioning (EKS clusters)** | Everything in read-only, plus EKS cluster and node-group lifecycle and management of the `ankra-eks-*` IAM roles Ankra creates for clusters. | You want to [provision EKS clusters](/guides/eks-clusters) from Ankra. |

Provisioning is a superset of read-only, so a single provisioning credential also covers cost and inventory. The access level is fixed when the credential is created - to move from read-only to provisioning later, connect again at the provisioning level (see [Changing access level](#changing-access-level)).

## Connect with an IAM role (recommended)

You will move between two browser tabs: the Ankra credential dialog and the AWS CloudFormation console. The whole flow takes about five minutes.

**Prerequisites:** an AWS account where you are allowed to create CloudFormation stacks and IAM roles (for example `AdministratorAccess`, or CloudFormation plus IAM write permissions).

<Steps>
  <Step title="Start the wizard in Ankra">
    Go to **Credentials** â†’ **Add** â†’ **AWS** and pick the access level - **Read-only (cost & inventory)** or **Provisioning (EKS clusters)**. Enter a credential name.

    The dialog shows your unique **external ID** (it starts with `ankra-`). This ID ties the role to your organisation and is pre-filled in the stack you are about to launch - you don't need to copy it anywhere.

    <Warning>
    Keep this dialog open until you finish. A fresh external ID is generated each time you open the wizard - if you close it and start again after creating the stack, the new ID won't match your stack and the connection will fail. See [troubleshooting](#troubleshooting) if this happens.
    </Warning>
  </Step>
  <Step title="Launch the CloudFormation stack">
    Click **Launch CloudFormation stack**. This opens the AWS console's quick-create page in a new tab, with everything pre-filled: the template, the stack name (`AnkraProvisioning` or `AnkraCloudCostReadOnly`), your external ID, and the Ankra principal that will assume the role.

    1. If prompted, sign in to the AWS account you want to connect.
    2. Leave the **ExternalId** and **AnkraPrincipalArn** parameters unchanged. You can rename the stack or role if your organisation requires it.
    3. Tick **I acknowledge that AWS CloudFormation might create IAM resources with custom names** at the bottom of the page.
    4. Click **Create stack**.
  </Step>
  <Step title="Wait for CREATE_COMPLETE">
    The stack creates a single IAM role and usually finishes in under a minute. Wait until the stack status shows **CREATE_COMPLETE** (use the refresh button on the stack page).
  </Step>
  <Step title="Copy the Role ARN from the Outputs tab">
    On the stack page, open the **Outputs** tab and copy the value of the **RoleArn** output. It looks like:

    ```
    arn:aws:iam::<account-id>:role/AnkraProvisioning
    ```

    If you don't see an Outputs tab, the stack hasn't finished creating yet, or you are looking at a different stack.
  </Step>
  <Step title="Paste the ARN into Ankra and test">
    Back in the Ankra dialog, paste the ARN into **Role ARN**. Set the **Default region** to the region you want Ankra to use for cost data and as the default for new EKS clusters.

    Click **Test connection**. Ankra performs a real `sts:AssumeRole` call against your new role - on success you'll see "AWS role assumed successfully".
  </Step>
  <Step title="Connect">
    Click **Connect AWS account**. Ankra validates the role once more and saves the credential. It now appears in your credentials list and can be selected when [creating an EKS cluster](/guides/eks-clusters) (provisioning level) or used automatically for cost and inventory sync.
  </Step>
</Steps>

### What the stack creates

The template creates one IAM role and nothing else. Its trust policy only allows Ankra's AWS principal to assume it, and only when the request carries your external ID, with sessions capped at one hour. Ankra never stores AWS keys for this credential - every operation assumes the role with short-lived credentials.

- **Read-only role** (`AnkraCloudCostReadOnly` by default): `pricing:*` read, `ce:GetCostAndUsage` / `ce:GetCostForecast`, `ec2:Describe*` inventory calls, and `eks:ListClusters` / `eks:DescribeCluster`.
- **Provisioning role** (`AnkraProvisioning` by default): the above, plus `eks:*`, and IAM permissions scoped to roles named `ankra-eks-*` - the roles Ankra creates for EKS clusters and node groups. The template also accepts an optional `PermissionsBoundaryArn` parameter if your organisation requires a permissions boundary on new roles.

To revoke Ankra's access at any time, delete the CloudFormation stack in AWS - this deletes the role, and no further access is possible. Then delete the credential in Ankra.

<Warning>
Deleting the stack while the credential is still in use breaks cost sync and, at the provisioning level, Ankra's ability to manage your EKS clusters. Delete or replace the Ankra credential first if clusters depend on it.
</Warning>

## Connect with access keys

<Steps>
  <Step title="Create an IAM user">
    In AWS IAM, create a user (or use an existing one) with permissions matching your use - read-only for cost and inventory (for example the AWS-managed `ReadOnlyAccess` and `Billing` policies), or EKS, EC2, and IAM permissions to provision EKS. Create an access key for the user.
  </Step>
  <Step title="Add to Ankra">
    Go to **Credentials** â†’ **Add** â†’ **AWS** â†’ **Access keys**, enter a name, and paste the access key ID and secret access key.
  </Step>
</Steps>

<Note>
Use the IAM role method where you can - it avoids storing long-lived secrets, and access is revocable on the AWS side by deleting the CloudFormation stack.
</Note>

## Changing access level

The access level is fixed when a credential is created, and there is no edit flow. To upgrade from read-only to provisioning:

1. Run the [IAM role flow](#connect-with-an-iam-role-recommended) again at the **Provisioning (EKS clusters)** level, under a new credential name.
2. Once connected, delete the old read-only credential and its CloudFormation stack.

## Troubleshooting

**"AWS denied the role assumption. Check the role trust policy and external ID."**

The `sts:AssumeRole` call was rejected. In order of likelihood:

- **The stack hasn't finished.** Wait for `CREATE_COMPLETE`, then test again.
- **The external ID doesn't match.** This happens if you closed and reopened the Ankra dialog after launching the stack - each wizard session generates a fresh ID, so the stack was created with an old one. Fix it in AWS: on the stack page choose **Update** â†’ **Use existing template**, replace the `ExternalId` parameter with the ID currently shown in the Ankra dialog, and update the stack. Alternatively, delete the stack and relaunch it from the current dialog.
- **The ARN is wrong.** Make sure you copied the `RoleArn` value from the stack's Outputs tab, not the stack ARN or another role. AWS reports a nonexistent role as the same access-denied error.
- **The trust policy was edited.** If someone changed the role's trust policy after creation, restore it by updating the stack.

**"Could not reach the AWS STS endpoint."**

A network problem between Ankra and `sts.<region>.amazonaws.com`. Check that the default region is a valid AWS region and try again.

**"Could not validate the AWS credentials..." when clicking Connect AWS account**

The same validation as **Test connection** runs on save, so the causes above apply. Run **Test connection** to get the more specific message.

**"An AWS credential with this name already exists."**

Credential names are unique per organisation. Pick a different name, or delete the old credential first if you are replacing it.
