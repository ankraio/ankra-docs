## What is Ankra CLI?

The Ankra CLI is a powerful command-line interface for interacting with the Ankra platform. It enables you to manage clusters, chat with AI about your infrastructure, browse Helm charts, manage credentials, deploy stacks, and automate platform operations directly from your terminal.

## Features

- **SSO Login** Browser-based authentication with PKCE
- **Profile Authentication** Manage authenticator apps, recovery codes, and passkeys/security keys
- **AI Chat** Interactive and one-shot AI-powered troubleshooting
- **Organisation Management** List, switch, create organisations, manage members
- **Cluster Management** List, select, reconcile, provision, deprovision, delete, and clone clusters
- **Cloud Cluster Lifecycle** Provider-agnostic `upgrade`, `scale`, `node-group`, and `k3s-versions` (Hetzner/OVH/UpCloud/DigitalOcean/Proxmox VE/Morpheus auto-detected)
- **Cluster Access (RBAC)** Grant, list, and revoke per-user kube-gateway access
- **Cloud SSH Keys** View, replace, and re-sync SSH key credentials attached to cloud clusters
- **kubectl Integration** Write `ankra-*` kubeconfig contexts and mint short-lived cluster tokens
- **Cluster Metrics** Run instant and range PromQL queries against the cluster's Prometheus
- **Kubernetes Resources** Browse pods, deployments, services, nodes, and other resources directly
- **Pod Logs** Stream and tail pod logs with namespace and container filtering
- **Stack Management** Create, list, rename, delete, clone between clusters, and view stack history
- **Addon Management** List, configure, update, and uninstall cluster addons
- **Operations & Steps** List executions, inspect their steps, retry terminal ones, and cancel running executions or steps
- **Agent Management** View agent status, manage tokens, trigger upgrades
- **Manifest Management** List and inspect cluster manifests
- **In-Cluster Helm** List and uninstall Helm releases running in the cluster
- **Helm Charts** Browse, search, and get details on available charts
- **Helm Registries** Manage organisation Helm registries and registry credentials
- **SOPS Encryption** Encrypt and decrypt manifest and addon configuration files
- **Hetzner Cloud** Provision and manage Hetzner Kubernetes clusters
- **OVH Cloud** Provision and manage OVH Kubernetes clusters
- **UpCloud** Provision and manage UpCloud Kubernetes clusters
- **DigitalOcean** Provision and manage DigitalOcean Kubernetes clusters
- **Proxmox VE** Provision and manage Kubernetes clusters on your own Proxmox VE infrastructure
- **Morpheus** Provision and manage Kubernetes clusters through Morpheus
- **Credentials** Manage cloud provider and Git credentials
- **API Tokens** Create, list, revoke, and delete API tokens
- **Support Requests** Open and track AI-reviewed support tickets from the terminal
- **Shell Completion** Generate completion scripts for bash, zsh, fish, and powershell
- **Agent Skills** Install the curated Ankra Agent Skills into Cursor or Claude Code (works offline)
- **Secure** Credentials stored locally in `~/.ankra.yaml`

## Installation

### Single Command Installation (Recommended)

```bash
bash <(curl -sL https://github.com/ankraio/ankra-cli/releases/latest/download/install.sh)
```

After installation, verify with:
```bash
ankra --version
```

### Direct Download

Download the binary directly from [GitHub Releases](https://github.com/ankraio/ankra-cli/releases/).

```bash
chmod +x ankra
mv ankra ~/bin/ankra  # or /usr/local/bin/ankra
```

### Upgrading the CLI

The CLI can update itself in place with `ankra upgrade` (aliased as `ankra self-update`). The downloaded binary is verified against its published SHA-256 checksum before it replaces the running executable.

```bash
# Upgrade to the latest stable release
ankra upgrade

# Check whether a newer release exists without installing
ankra upgrade --check

# Pin an exact version (also works as a downgrade / rollback)
ankra upgrade --version v0.3.0
```

| Flag | Description |
|------|-------------|
| `--version <tag>` | Install an exact release (e.g. `v0.3.0` or `0.3.0`); installs whether newer or older, so it doubles as a rollback |
| `--check` | Report whether a newer release is available without installing |
| `--beta` | Include pre-release versions for this run (overrides the saved channel) |
| `--force` | Reinstall even if already on the target version |
| `-y, --yes` | Skip the confirmation prompt |
| `--allow-unverified` | Install even when no checksum is published (insecure) |

<Tip>
If the binary lives in a root-owned location, run `sudo ankra upgrade` or re-run the install script.
</Tip>

#### Beta (pre-release) channel

Opt in to release candidates (such as `v0.3.0-rc.1`) so `ankra upgrade` resolves the newest release including pre-releases:

```bash
ankra config beta enable    # opt in
ankra config beta status    # show current channel
ankra config beta disable   # back to stable only
```

The channel preference is stored locally in `~/.ankra/settings.json` (separate from your credentials).

## Authentication

### SSO Login (Recommended)

The easiest way to authenticate is using browser-based SSO:

```bash
ankra login
```

This will:
1. Open your browser to the Ankra login page
2. After you authenticate, ask you to approve the sign-in in the browser — the approval page names the machine that ran `ankra login`
3. If your account requires MFA, continue into the browser challenge and wait for you to complete the second step
4. Save your credentials locally, after which you can use all ankra CLI commands

The whole login happens between your browser and the platform — the CLI never opens a local network port. Only approve a sign-in that names a machine you just ran `ankra login` on.

Your credentials are saved to `~/.ankra.yaml`.

To logout and clear credentials:
```bash
ankra logout
```

### Environment Variables (Alternative)

You can also use environment variables:

```bash
export ANKRA_API_TOKEN=your-token-here
export ANKRA_BASE_URL=https://platform.ankra.app  # optional
```

Add to your shell profile (`~/.zshrc` or `~/.bashrc`) for persistence.

## Quick Start

```bash
# 1. Login with SSO
ankra login

# 2. List your clusters
ankra cluster list

# 3. Select a cluster to work with
ankra cluster select

# 4. Start chatting with AI about your infrastructure
ankra chat "What's the status of my deployments?"
```

## Machine-readable output (`-o json|yaml`)

Commands that read or return data support structured output via the shared `-o`/`--output` flag, so scripts and AI agents never have to parse tables or prose:

- `-o json` prints the result as indented JSON; `-o yaml` emits the same data as YAML.
- The kubectl-style commands (`cluster get ...`, `cluster helm releases`, `cluster metrics ...`) default to `-o table` and accept `json`/`yaml` as alternatives.

```bash
ankra cluster list -o json
ankra cluster operations list -o json
ankra cluster get pods -o json
ankra org list -o json
```

Write commands (reconcile, provision, deprovision, scaling, node groups, token creation, …) also accept `-o json` and emit the API result — including operation IDs — so automation can poll `ankra cluster operations list <id> -o json` for completion. Asynchronous writes submitted without `--wait` emit `{"submitted": true, ...}`.

### Profile Authentication

Manage your own two-factor authentication settings from the CLI. Passkey and security-key creation opens **Profile → Authentication** in the browser because WebAuthn registration must happen in a browser, but the CLI can list and remove existing passkeys.

#### profile auth status
Show whether two-factor authentication is required, whether you are enrolled, how many recovery codes remain, and which passkeys/security keys are registered.

```bash
ankra profile auth status
ankra profile auth status -o json
```

#### profile auth totp
Set up or remove an authenticator app.

```bash
ankra profile auth totp start
ankra profile auth totp confirm <6-digit-code>
ankra profile auth totp remove --yes
```

`totp start` prints an `otpauth://` URI and manual secret for your authenticator app. `totp confirm` may print one-time recovery codes; save them immediately because they are not shown again.

#### profile auth recovery-codes
Generate a fresh set of one-time recovery codes. This invalidates the existing set.

```bash
ankra profile auth recovery-codes regenerate
ankra profile auth recovery-codes regenerate --yes
ankra profile auth recovery-codes regenerate -o json --yes
```

#### profile auth passkeys
List or remove passkeys/security keys, or open the browser page to add one.

```bash
ankra profile auth passkeys list
ankra profile auth passkeys remove <credential_id> --yes
ankra profile auth passkeys open
```

### Organisation Management

**Aliases:** `ankra org`, `ankra organisation`, `ankra organization`

#### org list
List all organisations you belong to.

```bash
ankra org list
```

#### org switch
Switch to a different organisation. This persistently changes your selected
organisation for all future commands.

```bash
ankra org switch <org_id>
```

#### Run a single command against another organisation

Use the global `--org` flag (organisation name or ID) to scope a single command
to a different organisation **without** changing your selected organisation. You
can also set the `ANKRA_ORG` environment variable to scope a whole shell session.

```bash
ankra --org "Acme Corp" cluster list                  # by name, just this command
ankra --org <org_id> cluster info my-cluster           # by ID, just this command

export ANKRA_ORG="Acme Corp"                           # for the whole session
ankra cluster list
```

The override is applied per request; it never calls `ankra org switch`. You must
be an active member of the target organisation, otherwise the API returns a
permission error.

#### org current
Show the currently selected organisation.

```bash
ankra org current
```

#### org create
Create a new organisation.

```bash
ankra org create <name>
ankra org create <name> --country <code>
```

#### org members
List members of an organisation.

```bash
ankra org members              # current organisation
ankra org members <org_id>     # specific organisation
```

#### org invite
Invite a user to the current organisation.

```bash
ankra org invite <email>
ankra org invite <email> --role operator
```

`--role` accepts `owner`, `admin`, `operator`, `member`, `viewer`, and `read-only`, validated client-side. Until the RBAC assignments API is fully live, `owner`/`operator` alias onto `admin`/`member` on the wire.

#### org roles
List the built-in roles with their descriptions, and define custom roles.

```bash
ankra org roles                                   # list roles
ankra org roles create --name release-manager \
  --permissions clusters.read,stacks.deploy \
  --kube-access edit                              # optional bundled kube access
```

#### org cluster-groups
Manage named sets of clusters used as role-assignment scopes. Groups are **static** (pinned members) or **dynamic** (a label selector evaluated against cluster labels).

```bash
ankra org cluster-groups list
ankra org cluster-groups create --name prod --selector env=production   # dynamic
ankra org cluster-groups create --name core                             # static
ankra org cluster-groups add-cluster core <cluster_id>
ankra org cluster-groups set-selector prod tier=production,region=eu
ankra org cluster-groups preview prod                                   # clusters it resolves to
```

#### org assign / assignments / unassign
Grant, list, and revoke scoped role assignments. A role can be assigned at organisation, single-cluster, or cluster-group scope; when the role bundles Kubernetes access, Ankra provisions it across every cluster in scope.

```bash
ankra org assign user@example.com --role operator --cluster-group prod
ankra org assign user@example.com --role viewer --cluster <cluster_id>
ankra org assign user@example.com --role admin                          # org-wide
ankra org assignments user@example.com                                  # list what a member holds
ankra org unassign <assignment_id>
```

#### org remove
Remove a user from the current organisation.

```bash
ankra org remove <user_id>
ankra org remove <user_id> -f   # skip confirmation
```

#### org variables
Manage organisation-scoped template variables. These are available to every
cluster in the organisation as substitutions in stack manifests and addon
values at deploy time, and are shadowed by cluster- or stack-scoped variables
of the same name.

```bash
ankra org variables list                                       # list all
ankra org variables get DB_HOST                                # show one
ankra org variables set DB_HOST db.example.com                 # upsert
ankra org variables set API_TOKEN -                            # value from stdin
ankra org variables delete DB_HOST                             # delete (prompts)
```

| Subcommand | Notable flags |
|------------|---------------|
| `list` | `-o json\|yaml` |
| `get` | `-o json\|yaml` |
| `set` | `--description <text>` (creates or updates; pass `-` for the value to read from stdin) |
| `delete` | `--yes` skips the confirmation prompt |

### Cluster Operations

#### cluster list
List all clusters.

```bash
ankra cluster list
```

#### cluster info
Show details of a specific cluster. Defaults to the currently selected cluster if no name is provided.

**Aliases:** `ankra cluster get-cluster`

```bash
ankra cluster info                  # uses selected cluster
ankra cluster info <cluster-name>   # specific cluster
```

#### cluster select
Select a cluster for subsequent commands. Supports both interactive selection and direct selection by name.

```bash
ankra cluster select                 # interactive picker
ankra cluster select <cluster-name>  # select directly by name
```

#### cluster clear
Clear the active cluster selection.

```bash
ankra cluster clear
```

#### Run one cluster command against another cluster

All `ankra cluster ...` subcommands inherit a `--cluster <name|id>` override. Use it to target a cluster for one command without changing the selected cluster saved by `ankra cluster select`.

```bash
ankra cluster --cluster production operations list
ankra cluster --cluster 62f4559a-a44d-46d7-aab3-a57c9dd6b4c6 addons list
```

#### cluster reconcile
Trigger cluster reconciliation to sync desired state with actual state.

```bash
ankra cluster reconcile                  # uses selected cluster
ankra cluster reconcile <cluster-name>   # specific cluster
```

#### cluster apply
Apply an ImportCluster YAML to the Ankra API.

```bash
ankra cluster apply -f <file>

# Validate the YAML locally (offline) without calling the API
ankra cluster apply -f <file> --dry-run

# Submit, then block until the platform finishes (with a timeout)
ankra cluster apply -f <file> --wait --timeout 15m
```

| Flag | Description |
|------|-------------|
| `--dry-run` | Run the **local** structural, dependency, and YAML checks only, without calling the API. For server-side checks (chart existence, secret detection), use [`cluster validate`](#cluster-validate) |
| `--wait` | Submit the change, then wait for the operation to finish and report success/failure (default: submit and return immediately) |
| `--timeout` | Maximum time to wait when `--wait` is set (default `10m`) |

<Note>
Writes are submitted **asynchronously** by default - the command returns as soon as the platform accepts the change, and the work continues in the background. Add `--wait` to block until completion. Avoid resubmitting the same change while it may still be running.
</Note>

#### cluster validate
Validate an ImportCluster YAML without applying it. Runs the local `--dry-run` checks first, then sends the spec to the Ankra API for server-side validation that the offline checks cannot do: chart existence in your connected Helm registries, plaintext Kubernetes `Secret` / unencrypted addon-value detection, and parent references resolved against an existing cluster's resources.

```bash
ankra cluster validate -f <file>

# Treat plaintext secrets as errors instead of warnings
ankra cluster validate -f <file> --strict-secrets

# Validate against an existing cluster's deployed resources
ankra cluster validate -f <file> --cluster <cluster_id>
```

Warnings (such as plaintext secrets) are printed but do not fail the command; the command exits non-zero only when validation finds errors, so it fits straight into CI.

#### cluster draft
Stage an ImportCluster YAML as drafts instead of applying it. The local `--dry-run` checks run first, then each stack in the file is saved as a resource draft you can review, edit, and deploy from the Ankra stack builder — nothing is deployed by the command.

```bash
ankra cluster draft -f <file>
```

If the cluster does not exist yet it is imported first (live), because drafts can only be attached to an existing cluster. Stacks that already match the cluster's desired state are reported as `no changes` rather than creating an empty draft, and the command exits non-zero if any stack fails validation.

#### cluster clone
Clone stacks from an existing cluster to a new cluster configuration.

```bash
# Clone all stacks
ankra cluster clone source-cluster.yaml new-cluster.yaml

# Clone from a remote URL
ankra cluster clone https://github.com/org/repo/raw/main/cluster.yaml new-cluster.yaml

# Clone only specific stacks
ankra cluster clone cluster.yaml new-cluster.yaml --stack "monitoring" --stack "networking"

# Replace all stacks in target
ankra cluster clone cluster.yaml new-cluster.yaml --clean

# Force merge even with conflicts
ankra cluster clone cluster.yaml new-cluster.yaml --force

# Copy missing files from skipped stacks
ankra cluster clone cluster.yaml new-cluster.yaml --copy-missing
```

| Flag | Description |
|------|-------------|
| `--clean` | Replace all stacks in the target (clean copy) |
| `--force` | Force merge even when names conflict |
| `--copy-missing` | Copy missing files even for skipped stacks |
| `--stack` | Clone only specific stacks by name (repeatable) |

#### cluster provision
Provision (start) a managed cluster.

```bash
ankra cluster provision                  # uses selected cluster
ankra cluster provision <cluster-name>   # specific cluster
```

#### cluster deprovision
Deprovision (stop) a managed cluster. Accepts a cluster name **or** ID, and falls back to the selected cluster. For cloud clusters (Hetzner, OVH, UpCloud, DigitalOcean) it routes to the provider-specific teardown endpoint so cloud resources are released.

```bash
ankra cluster deprovision                             # uses selected cluster
ankra cluster deprovision <cluster-name|id>           # specific cluster
ankra cluster deprovision <cluster-name|id> --auto-delete  # auto-delete after deprovision
ankra cluster deprovision <cluster-name|id> --force        # force even if unexpected state
```

| Flag | Description |
|------|-------------|
| `--auto-delete` | Automatically delete the cluster after deprovisioning |
| `--force` | Force deprovision even if cluster is in an unexpected state |

#### cluster roll-to
Roll a cluster resource to a specific version.

```bash
ankra cluster roll-to --version <resource_version_id>
ankra cluster roll-to --version <resource_version_id> --cluster <cluster_id>
```

| Flag | Description |
|------|-------------|
| `--version` | Resource version ID to roll to (required) |
| `--cluster` | Cluster ID (defaults to selected cluster) |

#### delete cluster
Delete a cluster by name.

```bash
ankra delete cluster <name>
ankra delete cluster <name> -f   # skip confirmation
```

#### cluster access

Manage who can reach a cluster's Kubernetes API through Ankra's API proxy (organisation admins only). A grant maps an organisation member to a Kubernetes role, cluster-wide or limited to one namespace. See [Cluster Access](/essentials/cluster-access) for roles, scopes, and how grants are reconciled into RBAC.

```bash
# List grants and their RBAC reconcile status
ankra cluster access list --cluster my-cluster

# Grant a member access (cluster-wide view by default)
ankra cluster access grant user@example.com --role view

# Limit a grant to one namespace
ankra cluster access grant user@example.com --role edit --namespace staging

# Revoke a single grant by ID, or every grant a member has on the cluster
ankra cluster access revoke 6f1f9aca-2c3d-4e5f-8a9b-0c1d2e3f4a5b
ankra cluster access revoke user@example.com
```

| Flag | Description |
|------|-------------|
| `--cluster` | Cluster name or ID (defaults to the selected cluster) |
| `--role` | Kubernetes role for the grant: `view` (default), `edit`, `admin`, or `cluster-admin` |
| `--namespace` | Limit the grant to one namespace (default: cluster-wide) |
| `-o, --output` | `json` or `yaml` for `list` and `grant` |

#### cluster kubeconfig

Manage the Ankra entries in your local kubeconfig so you can run `kubectl` against a cluster through Ankra's API proxy. See [Accessing Clusters with kubectl](/essentials/kubeconfig) for the full flow.

```bash
# Add the selected cluster and make it active
ankra cluster kubeconfig add --cluster my-cluster --use

# Add every cluster you can access
ankra cluster kubeconfig add --all

# List and remove Ankra-managed contexts
ankra cluster kubeconfig list
ankra cluster kubeconfig remove --cluster my-cluster
```

By default `add` writes an exec-based context that fetches a short-lived token on demand (auto-refreshing, SSO-backed). Use `--embed-token` for a self-contained file, `--print` to write a standalone kubeconfig to stdout, and `--namespace` to set a default namespace. Other entries in your kubeconfig are left untouched.

#### cluster kube-token

Print a short-lived Kubernetes `ExecCredential` JSON. This is invoked by `kubectl` as a credential plugin (the default `cluster kubeconfig add` wires it up for you); you rarely run it directly.

```bash
ankra cluster kube-token --cluster <cluster-name-or-id>
```

#### cluster metrics

Query the Prometheus metrics source configured for a cluster (Cluster Settings → Metrics), proxied through the agent. See [Prometheus integration](/integrations/prometheus).

```bash
# Instant PromQL query
ankra cluster metrics query 'up'
ankra cluster metrics query 'sum(rate(container_cpu_usage_seconds_total[5m])) by (pod)'

# Range query
ankra cluster metrics query-range 'rate(node_cpu_seconds_total[5m])' --range 6h --step 5m
```

| Flag | Description |
|------|-------------|
| `--cluster` | Cluster name (defaults to the selected cluster) |
| `--range` | Duration relative to now for `query-range` (e.g. `15m`, `1h`, `6h`, `24h`, `7d`) |
| `--start` / `--end` | Range bounds as Unix seconds (alternative to `--range`) |
| `--step` | Resolution step for `query-range` (e.g. `30s`, `1m`, `5m`) |
| `--timeout` | Query timeout in seconds |
| `-o, --output` | `table` (default), `json`, or `yaml` |

### SOPS Encryption

Encrypt and decrypt manifest and addon configuration values using SOPS.

All `encrypt` and `decrypt` subcommands support two modes:

- **Cluster mode (default)** — fetch the resource from a live cluster, encrypt
  or decrypt via the SOPS API, and (for encrypt) push the result back via the
  partial-stack PATCH endpoint along with `encrypted_paths`. No local files
  needed. The owning stack is resolved automatically.
- **File mode (`-f cluster.yaml`)** — rewrite the local `from_file` referenced
  by a `cluster.yaml`, updating `encrypted_paths` in the YAML. Used by GitOps
  workflows where the source of truth lives on disk.

The two modes are mutually exclusive. Without `-f`, the CLI uses the selected
cluster (or the cluster passed via `--cluster <name|id>`).

#### cluster sops-config
Display the SOPS encryption configuration including the public key for the organisation.

```bash
ankra cluster sops-config
```

#### cluster encrypt manifest
Encrypt a specific key in a manifest.

```bash
# Cluster mode (selected cluster)
ankra cluster encrypt manifest <manifest_name> --key <key_name>

# Cluster mode (specific cluster)
ankra cluster encrypt manifest <manifest_name> --key <key_name> --cluster <name|id>

# File mode
ankra cluster encrypt manifest <manifest_name> --key <key_name> -f <cluster.yaml>
```

| Flag | Description |
|------|-------------|
| `--key` | YAML key name to encrypt (e.g. `password`, not `data.password`) — required. A dotted value is normalised to its last segment with a notice |
| `--cluster` | Target cluster (name or ID); defaults to the active selection (cluster mode) |
| `-f`, `--file` | Path to a local `cluster.yaml` (enables file mode) |

#### cluster encrypt addon
Encrypt a specific key in an addon's Helm values.

```bash
# Cluster mode
ankra cluster encrypt addon --name <addon_name> --key <key_name> [--cluster <name|id>] [--stack <stack>]

# File mode
ankra cluster encrypt addon --name <addon_name> --key <key_name> -f <cluster.yaml>
```

| Flag | Description |
|------|-------------|
| `--name` | Addon name — required |
| `--key` | YAML key name to encrypt (e.g. `adminPassword`) — required. A dotted value is normalised to its last segment with a notice |
| `--cluster` | Target cluster (cluster mode) |
| `--stack` | Required (cluster mode) when the addon exists in multiple stacks |
| `-f`, `--file` | Path to a local `cluster.yaml` (enables file mode) |

#### cluster decrypt manifest
Decrypt and print a manifest. Cluster mode fetches from the live cluster; file
mode reads the referenced `from_file` from a local `cluster.yaml`.

```bash
ankra cluster decrypt manifest <manifest_name> [--cluster <name|id>]
ankra cluster decrypt manifest <manifest_name> -f <cluster.yaml>
```

#### cluster decrypt addon
Decrypt and print an addon's Helm values. Cluster mode fetches from the live
cluster; file mode reads the referenced `from_file` from a local `cluster.yaml`.

```bash
ankra cluster decrypt addon --name <addon_name> [--cluster <name|id>] [--stack <stack>]
ankra cluster decrypt addon --name <addon_name> -f <cluster.yaml>
```

<Tip>
You can add new encrypted keys to files that are already SOPS-encrypted, as long as they were encrypted with your organisation's key.
</Tip>

<Warning>
If a file was encrypted by a different organisation, you'll need to decrypt it first using the original key, then encrypt it with your organisation's key.
</Warning>

```bash
ankra cluster stacks delete <stack-name>
```

#### cluster stacks rename

```bash

```

#### cluster stacks history
Show change history for a stack.

```bash
ankra cluster stacks history <stack-name>
```

#### cluster stacks clone
Clone a stack to another cluster as a draft.

```bash
ankra cluster stacks clone <stack-name> --to <target-cluster>
ankra cluster stacks clone my-stack --to production --name my-stack-prod
ankra cluster stacks clone my-stack --to production --include-config=false
```

| Flag | Short | Description |
|------|-------|-------------|
| `--to` | `-t` | Target cluster name or ID (required) |
| `--name` | `-n` | Custom name for the cloned stack |

| `--dry-run` | Print the before/after spec without applying |
| `-o json\|yaml` | Machine-readable output |

#### cluster manifests upgrade
Upgrade a manifest in-place. Replace the whole manifest from a file or stdin, or
patch a single path inside the Kubernetes YAML with helm-style `--set` (for


| Flag | Description |
|------|-------------|
| `--set` | Mutate one path in the manifest YAML (repeatable; supports `containers[0]` and `containers[name=app]` selectors) |
| `--set-string` | Like `--set` but always treats the value as a string |
| `--set-file` | Like `--set` but reads the value from a file (`key=path` or `key=@path`) |
| `--target-kind` | With `--set`: pick the document by Kubernetes kind (multi-doc manifests) |
| `--target-name` | With `--set`: pick the document by `metadata.name` (multi-doc manifests) |
| `--add-parent` | Add a dependency parent `name=<n>,kind=<manifest\|addon>` (kind defaults to `manifest`; repeatable) |
| `--remove-parent` | Remove a dependency parent (repeatable) |
| `--set-parent` | Replace ALL dependency parents with the given set (repeatable) |
| `--from-file` | Replace the entire manifest from a file |
| `--manifest -` | Replace the entire manifest from stdin |
| `--namespace` | Change the manifest's namespace |
| `--dry-run` | Print the before/after spec without applying |
| `-o json\|yaml` | Machine-readable output |

`--set*` MUTATE the existing manifest and are mutually exclusive with `--from-file` / `--manifest -`, which REPLACE it. A list selector that matches nothing fails with a clear error rather than guessing. Parent flags (`--add-parent` / `--remove-parent` / `--set-parent`) edit dependency parents; `--set-parent` replaces the list wholesale and cannot be combined with the others.

### Connect kubectl

Wire `kubectl` straight to an Ankra-managed cluster through the kube gateway. `kubeconfig add` writes an `ankra-*` context into your kubeconfig with atomic `0600` writes that leave foreign clusters, users, and contexts untouched. By default it uses a client-go exec credential plugin that calls `cluster kube-token` to fetch a short-lived, SSO-backed token on demand — run `ankra login` once first.

#### cluster kubeconfig add
Add or update an Ankra context. Writes to `--kubeconfig` if given, otherwise the first entry of `$KUBECONFIG`, otherwise `~/.kube/config`.

```bash
ankra cluster kubeconfig add --cluster my-cluster --use         # exec-based context, set as current
ankra cluster kubeconfig add --all                              # every cluster you can access
ankra cluster kubeconfig add --cluster my-cluster --embed-token # embed a short-lived token instead of exec
ankra cluster kubeconfig add --cluster my-cluster --print > my-cluster.yaml  # standalone file, don't touch ~/.kube/config
```

| Flag | Description |
|------|-------------|
| `--cluster` | Cluster name or ID (defaults to the selected cluster) |
| `--all` | Add every cluster you can access |
| `--embed-token` | Embed a short-lived token instead of the exec credential plugin |
| `--namespace` | Default namespace for the context |
| `--use` | Set the added context as the active `current-context` |
| `--print` | Print a standalone kubeconfig to stdout instead of writing the file |
| `--kubeconfig` | Path to the kubeconfig file (default: `$KUBECONFIG` or `~/.kube/config`) |
| `--exec-command` | Executable kubectl invokes for credentials in exec mode (default: `ankra`) |
| `--insecure-skip-tls-verify` | Skip TLS verification for the cluster server (dev only) |

#### cluster kubeconfig list
List the Ankra-managed contexts in your kubeconfig.

```bash
ankra cluster kubeconfig list
```

#### cluster kubeconfig remove
Remove Ankra-managed contexts.

```bash
ankra cluster kubeconfig remove --cluster my-cluster
ankra cluster kubeconfig remove --all
```

#### cluster kube-token
Print a short-lived Kubernetes `ExecCredential` (JSON) for use as a kubeconfig credential plugin. This is normally invoked by `kubectl`, not by hand; `kubeconfig add` wires it up for you. It never prompts — run `ankra login` first.


#### cluster upgrade
Upgrade the Kubernetes version on all nodes of a cloud cluster, on both k3s and kubeadm distributions. Only Hetzner, OVH, UpCloud, DigitalOcean, Proxmox VE, and Morpheus clusters can be upgraded.

```bash
ankra cluster upgrade <cluster_id> <target_version>
ankra cluster upgrade 62f4559a-a44d-46d7-aab3-a57c9dd6b4c6 v1.36.1+k3s1
```

On kubeadm clusters, nodes upgrade one at a time (control plane first): each is cordoned, drained respecting PodDisruptionBudgets, upgraded, and gated on being Ready at the target version, with an etcd snapshot taken before the control plane. Add `--force` to proceed when a drain is blocked by a PodDisruptionBudget (the default aborts safely). The command prints the operation ID - follow it with `ankra cluster operations list`.

#### cluster scale
Scale the default worker pool of a cloud cluster up or down. To scale a named pool instead, use `cluster node-group scale`.

```bash
ankra cluster scale <cluster_id> <worker_count>
ankra cluster scale 62f4559a-a44d-46d7-aab3-a57c9dd6b4c6 3
```

#### cluster node-group
List, add, scale, upgrade, and delete node groups on a cloud cluster.

```bash
ankra cluster node-group list <cluster_id>
ankra cluster node-group add <cluster_id> --name <name> --instance-type <type> --count <n>
ankra cluster node-group scale <cluster_id> <group_name> <count>
ankra cluster node-group upgrade <cluster_id> <group_name> <instance_type>
ankra cluster node-group delete <cluster_id> <group_name>
```

The `add`, `scale`, `upgrade`, and `delete` mutations submit asynchronously by default (return on `202 Accepted`); add `--wait` to block until the platform finishes, bounded by `--timeout` (default 10m).

<Note>
OVH-specific node label and taint management (`node-group labels` / `node-group taints`) has no provider-agnostic equivalent yet — use `cluster ovh node-group labels|taints` for those.
</Note>

#### cluster node-group autoscaling
Read and write a node group's Cluster Autoscaler settings on Hetzner, OVH, and UpCloud clusters. Enabling keeps the group's node count within `[min, max]` based on pod demand; the first enable on a cluster installs the autoscaler.

```bash
ankra cluster node-group autoscaling get <cluster_id> <group_name>
ankra cluster node-group autoscaling set <cluster_id> <group_name> --enabled --min 2 --max 10
ankra cluster node-group autoscaling set <cluster_id> <group_name> --enabled=false
```

Both honour `-o json|yaml`, and `set` supports the standard `--wait`/`--timeout` async-write flags.

#### cluster ssh-keys
View, replace, or re-sync the SSH key credentials attached to a Hetzner, OVH, UpCloud, DigitalOcean, Proxmox VE, or Morpheus cluster. The provider is detected from the cluster.

```bash
ankra cluster ssh-keys get <cluster_id>
ankra cluster ssh-keys set <cluster_id> --ssh-key-credential-ids <id>,<id>
ankra cluster ssh-keys set <cluster_id> --clear
ankra cluster ssh-keys resync <cluster_id>
```

| Flag | Description |
|------|-------------|
| `--ssh-key-credential-ids` | SSH key credential IDs, comma-separated or repeatable |
| `--clear` | Remove all user SSH keys; the Ankra-managed key is retained |
| `-o, --output` | `json` or `yaml` |

### Agent Skills

Install the curated [Ankra Agent Skills](https://github.com/ankraio/ankra-skills) into a
Cursor or Claude Code skills directory so your AI agent follows Ankra's recommended practices for the
CLI, ImportCluster YAML, stacks/addons, GitOps, CI/CD, SOPS secrets, Helm registries,
observability, alerts, Terraform, and cloud clusters. The skills are embedded in the CLI, so
installation works offline.

```bash
ankra skills list                            # list available skills (marks installed ones)
ankra skills install                         # install all into ~/.cursor/skills (personal)
ankra skills install --editor claude-code    # install all into ~/.claude/skills
ankra skills install --project .             # install into ./.cursor/skills (project)
ankra skills install --editor claude-code --project .  # install into ./.claude/skills
ankra skills install ankra-cli ankra-gitops  # install only named skills
ankra skills install --force                 # overwrite existing skills
ankra skills install --source ./skills       # install from a local source directory
ankra skills uninstall                       # remove all Ankra skills
ankra skills uninstall ankra-cli             # remove a named skill
```

`--source <dir>` is also accepted by `skills list` and `skills uninstall`, which is useful for testing a local copy of the curated skill set instead of the embedded copy. Use `--editor cursor` (default) or `--editor claude-code` to choose the target skills directory.

#### Making Ankra the agent's default

Skills alone only load when the conversation happens to match their description. By default, `ankra skills install` also writes an **always-applied rule** telling your agent that clusters here are Ankra-managed: route changes through the GitOps repo or `ankra cluster apply`, inspect freely, and never mutate with raw kubectl/helm. Cursor gets a local plugin rule (`~/.cursor/plugins/local/ankra`) or a project `.cursor/rules/ankra.mdc`; Claude Code gets a marker-managed block in `CLAUDE.md`. Pass `--no-rules` to skip it; a full `ankra skills uninstall` cleans everything up.

```bash
ankra skills install --with-hooks   # also install a workflow-enforcing agent hook
ankra skills install --no-rules     # skills only, no always-applied rule
```

`--with-hooks` installs an agent hook (Cursor `beforeShellExecution`, Claude Code `PreToolUse`) that runs the `ankra skills guard` plumbing command on every shell command: cluster-mutating kubectl/helm invocations (`apply`, `delete`, `helm upgrade`, …) pause for confirmation with a redirect to the Ankra workflow, while read-only inspection, `--dry-run`, and everything else pass through untouched. The guard fails open and merges into existing `hooks.json`/`settings.json` without disturbing other entries.

<Note>
`ankra skills` installs the static, curated skill set. To generate a SKILL.md describing one
of your own clusters, use `ankra openclaw skill` instead.
</Note>

## Configuration

The CLI stores configuration in `~/.ankra.yaml`:

```yaml
token: your-api-token
base-url: https://platform.ankra.app
token_id: token-uuid
token_name: ankra-cli-hostname-username
```

Token resolution priority (highest to lowest):
1. `--token` flag (explicit CLI argument)
2. Config file token (saved by `ankra login`)
3. `ANKRA_API_TOKEN` environment variable

The base URL can be overridden with `--base-url` or the `ANKRA_BASE_URL` environment variable.

## Troubleshooting

### Common Issues

**PATH issues:**
```bash
which ankra  # verify installation location
echo $PATH   # check PATH includes ankra location
```

**Authentication errors:**
```bash
ankra logout  # clear credentials
ankra login   # re-authenticate
```

**Check version:**
```bash
ankra --version
```

**Get help:**
```bash
ankra --help
ankra cluster --help
ankra chat --help
```

### Error Messages

- **"No active cluster selected"** Run `ankra cluster select` first
- **"Resource not found"** Verify the resource name and selected cluster
- **"Unauthorized"** Re-authenticate with `ankra login`

### Exit codes

The CLI returns a stable, documented exit code so scripts can branch on the failure class without parsing error text:

| Code | Meaning |
|------|---------|
| `0` | Success |
| `1` | API / runtime error |
| `2` | Usage or flag error (unknown command/flag, bad arguments, missing required flags) |
| `3` | Targeted resource not found |
| `4` | Confirmation declined |
| `5` | `--wait` / `--timeout` expiry on an asynchronous write |
| `6` | Authentication failure (missing/expired/rejected credentials, 401/403) - re-authenticate |
| `7` | RBAC permission denied (403 `permission_denied`) - your role lacks a permission; the CLI names it and points at an organisation admin |
