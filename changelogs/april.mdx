# April 2026 Release: UpCloud & OVH Cloud Integration, AI Insights Overhaul, Command Palette Upgrade & Log Viewer Redesign

This release adds full UpCloud and OVH Cloud support, overhauls AI Insights with a dedicated detail page, analytics dashboard, adaptive scanning, and platform-aware remediation, upgrades the Command Palette with direct links to every Kubernetes resource type, and redesigns the Log Viewer for large clusters with high-volume streams.

## OVH Cloud Cluster Provisioning

Provision fully managed Kubernetes clusters on OVH Cloud with configurable control planes, workers, networking, and gateway nodes.

**Infrastructure Components**:
- Private network with configurable VLAN, subnet CIDR, and DHCP range
- Gateway instance for secure SSH access to cluster nodes
- Control plane and worker instances with selectable OVH flavors
- SSH key deployment to all instances
- k3s distribution with optional version pinning

**Region Support**: Deploy to any OVH Cloud region including Gravelines (GRA7/GRA9/GRA11), Strasbourg (SBG5), Beauharnois (BHS5), Warsaw (WAW1), Frankfurt (DE1), London (UK1), Singapore (SGP1), and Sydney (SYD1).

**Cluster Lifecycle**:
- Create clusters with full configuration control
- Scale workers up or down (1–10 nodes)
- Upgrade Kubernetes versions (one minor version at a time)
- Deprovision clusters with automatic OVH resource cleanup

[OVH Clusters Guide →](/essentials/ovh-clusters)

## OVH API Credentials

Securely store OVH Cloud API credentials (application key, application secret, consumer key, and project ID) with automatic validation against the OVH API on creation.

**Credential Validation**: Ankra validates your OVH credentials by calling the OVH API during creation. Clear error messages are returned for permission issues (403) and invalid credentials (401).

**Generate Credentials**: Create your OVH API credentials at [api.ovh.com/createToken/](https://api.ovh.com/createToken/) with GET, POST, PUT, DELETE rights on `/cloud/project/*`.

[Credentials Guide →](/integrations/credentials#ovh-api-credentials)

## CLI v0.1.127: OVH Commands

The Ankra CLI now includes full OVH support with commands for both credentials and cluster management.

### OVH Credential Commands

```bash
# List OVH API credentials
ankra credentials ovh list

# Create an OVH API credential (prompts for secrets)
ankra credentials ovh create --name my-ovh-cred --project-id <project-id>

# List SSH key credentials
ankra credentials ovh ssh-key list

# Create/generate an SSH key credential
ankra credentials ovh ssh-key create --name my-key --generate
```

### OVH Cluster Commands

```bash
# Create an OVH cluster
ankra cluster ovh create \
  --name my-cluster \
  --credential-id <ovh-credential-id> \
  --ssh-key-credential-id <ssh-key-credential-id> \
  --region GRA7 \
  --control-plane-count 1 \
  --control-plane-flavor-id b2-15 \
  --worker-count 2 \
  --worker-flavor-id b2-15

# Deprovision a cluster
ankra cluster ovh deprovision <cluster_id>

# Check worker count
ankra cluster ovh workers <cluster_id>

# Scale workers
ankra cluster ovh scale <cluster_id> 4

# Check Kubernetes version
ankra cluster ovh k8s-version <cluster_id>

# Upgrade Kubernetes version
ankra cluster ovh upgrade <cluster_id> v1.35.1+k3s1
```

## AI Insights Overhaul

A major upgrade to Proactive AI Insights with a dedicated detail page, analytics dashboard, advanced filtering, adaptive scanning, and platform-aware remediation.

### Insight Detail Page

Each insight now has a full-page view with four tabs:

- **Overview**: Root cause analysis with summary, severity, confidence score, affected resources, and conversation starters for AI chat
- **Remediation**: Copy-pasteable remediation commands and platform-aware actions (stack changes, add-on configuration, resource navigation)
- **Health**: Health snapshot and anomaly data captured at time of detection
- **History**: Previous resolutions with effectiveness tracking, resolution type, and context

Navigate to related Kubernetes resources directly from the affected resources list — pods, deployments, services, and more are linked back to their resource pages.

### Analytics Dashboard

New analytics tab with configurable time ranges (7, 30, or 90 days):

| Metric | Description |
|--------|-------------|
| MTTR | Mean time to resolution across resolved insights |
| Severity Trend | Daily breakdown of new insights by severity level |
| Namespace Breakdown | Issue distribution across Kubernetes namespaces |
| Category Breakdown | Clickable chart that filters the insights list |
| Trend Direction | Improving, degrading, or stable indicators |

### Dashboard View

A new dashboard tab on the insights page with a severity donut chart, mini trend chart, and category breakdown at a glance.

### Advanced Filtering

- Filter insights by namespace, category, and date range
- Three view modes: cards, compact list, and grouped (by category or namespace)
- Root cause summary shown inline on insight cards

### Related Insights

Insights sharing the same namespace or category are linked on the detail page, helping you spot patterns across related issues.

### Adaptive Scanning

The maintenance scheduler now adapts scan frequency based on cluster health:

| Cluster State | Scan Interval |
|---------------|---------------|
| Critical issues active | Every 60 seconds |
| Active non-critical issues | Every 3 minutes |
| Healthy | Every 10 minutes |

### Platform-Aware Remediation

AI analysis now suggests platform-native actions alongside kubectl commands:

- **Stack changes**: Adjust Helm values or stack configuration
- **Add-on configuration**: Modify add-on settings
- **Resource navigation**: Jump to the affected resource in the dashboard
- **AI chat**: Pre-built prompts to continue investigating with the AI Assistant

Remediation commands and platform actions are stored per-insight and displayed on the Remediation tab.

### Resolution Tracking

Resolving an insight now captures:

- **Resolution type**: How the issue was resolved (manual fix, configuration change, etc.)
- **Resolution context**: Additional metadata about the fix
- **Health snapshot at resolution**: The cluster health state when the issue was resolved

When an issue recurs, previous resolutions include effectiveness data (e.g. "did not hold") so you can see if a past fix was durable.

### RAG Learning Loop

Marking an insight as "helpful" indexes its root cause analysis and recommendations into the RAG knowledge base. This feeds back into future AI analysis, improving the quality of root cause diagnosis and remediation suggestions over time.

### Keyboard Shortcuts

Navigate and manage insights entirely from the keyboard:

| Shortcut | Action |
|----------|--------|
| `j` / `k` | Navigate between insights |
| `Enter` | Open detail page |
| `p` | Quick peek (drawer) |
| `a` | Acknowledge |
| `r` | Resolve |
| `d` | Dismiss |
| `c` | Ask AI |
| `/` | Focus search |
| `1`–`4` | Switch tabs |
| `v` | Cycle view mode |
| `?` | Show keyboard shortcuts help |

### AI Insights API

| Endpoint | Method | Description |
|----------|--------|-------------|

## Command Palette — Direct Links to All Kubernetes Resources

The Command Palette (`⌘+K` / `Ctrl+K`) now navigates directly to every Kubernetes resource type in your cluster — not just Ankra pages. Previously the palette linked to a handful of resource categories. Now it provides direct navigation to 33 individual resource kinds.

### 33 Kubernetes Resource Types

**Workloads**: Pods, Deployments, StatefulSets, DaemonSets, ReplicaSets, Jobs, CronJobs, PodDisruptionBudgets, HorizontalPodAutoscalers

**Networking**: Services, Ingresses, Ingress Classes, Endpoints, NetworkPolicies, CSIDrivers, CSINodes, CSIStorageCapacities

**Configuration**: ConfigMaps, Secrets, Namespaces, Events

**Storage**: PersistentVolumes, Persistent Volume Claims, Storage Classes, VolumeAttachments

**RBAC**: Roles, RoleBindings, Cluster Roles, Cluster Role Bindings, Service Accounts, LimitRanges

**Other**: Custom Resource Definitions (CRD), Helm Releases, Nodes

### New Cluster Pages in the Palette

The palette now includes cluster-specific pages that were previously only accessible via the sidebar:

| Command | Description |
|---------|-------------|
| `Logs` | Log explorer for pod and container logs |
| `Metrics` | CPU, memory, network, and disk metrics |
| `Resources` | Kubernetes resources overview |
| `Cluster - General Settings` | Cluster configuration |
| `Cluster - GitOps Settings` | GitHub and GitOps integration |

### New Actions

| Command | Description |
|---------|-------------|
| `Keyboard Shortcuts` | Show keyboard shortcuts reference |
| `Base64 Encode` | Encode text to Base64 |
| `Base64 Decode` | Decode Base64 to text |
| `Documentation` | Open Ankra docs in a new tab |
| `Switch Organisation` | Switch between organisations |

### Fuzzy Search

All resources support fuzzy search with keyword aliases. Type "dep" to find Deployments, "hpa" for HorizontalPodAutoscalers, "pv" for PersistentVolumes, "rbac" to find Roles and RoleBindings, or "crd" for Custom Resource Definitions.

## Log Viewer Redesign — High-Performance Streaming for Large Clusters

The Log Viewer has been rebuilt from the ground up to handle clusters with hundreds of pods and high-volume log streams.

### Scalable Pod Loading

Previously, the Cluster Logs view fetched all pods in a single request capped at 500. Clusters with more pods would silently truncate results, showing "0 pods" for workloads whose pods fell outside the cap.

Pods are now fetched on-demand per namespace when you select a workload, with a much higher limit. The workload list and pod list are loaded independently, so the initial page load is faster and pod counts are always accurate.

### Accurate Pod-to-Workload Matching

Pod-to-workload matching previously used a name prefix heuristic (`pod.name.startsWith(workload.name)`) which failed for many real-world pod naming patterns. Matching now uses Kubernetes label selectors — the same mechanism Kubernetes itself uses to associate pods with their owning workload.

### Scroll Stability

The log viewer no longer forces auto-scroll when new logs arrive. When you scroll up to investigate older logs, your scroll position is preserved even under high-throughput streams (logs arriving every millisecond). A "scroll to bottom" button appears to resume following. Auto-follow only re-engages when you explicitly click the button or resume streaming.

### Live Search Filtering

Search and log level filters now apply to incoming log lines in real-time. Previously, typing in the search bar would filter existing logs but new streaming lines would bypass the filter and appear regardless. Filters are now checked against every incoming line before it reaches the display.

### Performance Under Load

Log lines are buffered and flushed to the display in batches (every 150ms) to prevent UI freezes during high-volume streams. The viewer uses virtualized rendering to efficiently handle tens of thousands of log lines without degrading scroll performance or memory usage.
